## 2. 정산관리 vs 파레트 정산 (구분)

이 프로젝트에는 **두 가지 정산 시스템**이 있습니다:

| 구분 | 정산관리 (`/settlements`) | 파레트 정산 (`/pallets`) |
|------|---------------------------|---------------------------|
| **용도** | 작업비·입출고비·택배비 등 **통합 정산** | **파레트 보관료** 전용 정산 |
| **테이블** | `settlements`, `settlement_files` | `pallet_monthly_settlements`, `pallet_fee_calculations` |
| **페이지** | `settlements.html` | `pallets.html` |
| **API** | `/api/settlements/*` | `/api/pallets/settlements/*` |

**정산관리**는 파레트 보관료를 포함해 작업비, 입출고비, 택배비, 특수작업, 오배송/누락 차감, 착불 택배비 등을 **한 화면에서 통합 관리**합니다.

## 3. 정산관리 DB 스키마

### settlements 테이블

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | SERIAL/INTEGER | PK |
| company_name | TEXT | 화주사명 |
| settlement_year_month | TEXT | 정산년월 (예: 2025-01) |
| work_fee | INTEGER | 작업비 |
| work_fee_file_url | TEXT | 작업비 파일 URL (Google Drive) |
| inout_fee | INTEGER | 입출고비 |
| inout_fee_file_url | TEXT | 입출고비 파일 URL |
| shipping_fee | INTEGER | 택배비 |
| shipping_fee_file_url | TEXT | 택배비 파일 URL |
| storage_fee | INTEGER | 보관료 (파레트 연동) |
| special_work_fee | INTEGER | 특수작업 비용 |
| error_deduction | INTEGER | 오배송/누락 차감 |
| collect_on_delivery_fee | INTEGER | 착불 택배비 |
| total_amount | INTEGER | 총액 |
| status | TEXT | 대기/전달/정산확인/입금완료 |
| tax_invoice_file_url | TEXT | 세금계산서 URL |
| memo | TEXT | 비고 |
| created_at, updated_at | TIMESTAMP | 생성/수정 시각 |

**UNIQUE**: `(company_name, settlement_year_month)`

### settlement_files 테이블

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | SERIAL/INTEGER | PK |
| settlement_id | INTEGER | FK → settlements |
| file_type | TEXT | work_fee, inout_fee, shipping_fee |
| file_name | TEXT | 파일명 |
| file_url | TEXT | Google Drive URL |
| file_size | INTEGER | 파일 크기 |
| uploaded_at | TIMESTAMP | 업로드 시각 |

## 4. 정산관리 API 엔드포인트

| 메서드 | 경로 | 설명 | 권한 |
|--------|------|------|------|
| GET | `/api/settlements/list` | 정산 목록 조회 | 관리자: 전체, 화주사: 본인만 |
| GET | `/api/settlements/<id>` | 정산 상세 조회 | 권한 검증 |
| POST | `/api/settlements/` | 정산 생성 (중복 시 UPDATE) | 관리자 |
| PUT | `/api/settlements/<id>` | 정산 수정 | 관리자 |
| DELETE | `/api/settlements/<id>` | 정산 삭제 | 관리자 |
| GET | `/api/settlements/data-sources` | 데이터 소스 조회 (파레트, 특수작업, C/S, 반품) | 관리자/화주사 |
| POST | `/api/settlements/upload-file` | 파일 업로드 (FormData/JSON) | 관리자 |
| POST | `/api/settlements/upload-file-start` | 청크 업로드 시작 | 관리자 |
| POST | `/api/settlements/upload-file-chunk` | 청크 업로드 | - |
| POST | `/api/settlements/upload-file-complete` | 청크 업로드 완료 | 관리자 |
| POST | `/api/settlements/<id>/delete-file` | 파일 삭제 | 관리자 |
| POST | `/api/settlements/<id>/delete-tax-invoice` | 세금계산서 삭제 | 관리자 |
| POST | `/api/settlements/bulk-update-status` | 일괄 상태 변경 | 관리자 |
| POST | `/api/settlements/<id>/upload-tax-invoice` | 세금계산서 업로드 | 관리자 |
| POST | `/api/settlements/<id>/update-status` | 상태 변경 (대기/전달/정산확인/입금완료) | 관리자/화주사(정산확인만) |
| GET | `/api/settlements/<id>/statement` | 정산 명세서 엑셀 다운로드 | 권한 검증 |

## 5. 데이터 소스 연동 (data-sources API)

정산 등록 시 **일괄 불러오기**로 다음 데이터를 가져옵니다:

| 소스 | 테이블/모듈 | 필드 | 비고 |
|------|-------------|------|------|
| 파레트 보관료 | `api.pallets.models.get_settlements` | storage_fee | 화주사명·정산월 매칭 |
| 특수작업 | `special_works` | special_work_fee | work_date 기준 월 필터 |
| 오배송/누락 | `customer_service` | error_wrong_delivery_count, error_missing_count | issue_type 필터 |
| 착불 택배비 | `returns` | collect_on_delivery_fee | shipping_fee "착불" 패턴 파싱 |

**화주사명 매칭**: `normalize_company_name()`, `get_company_search_keywords()` 사용 (대소문자·띄어쓰기 무시)

## 6. 정산관리 프론트엔드 구조 (settlements.html)

### 주요 섹션

1. **정산 등록/수정** (관리자 전용)
   - 화주사 검색 가능 선택 (st-searchable-select)
   - 정산년월 (month input)
   - 일괄 불러오기 (data-sources API)
   - 금액 필드: 작업비, 입출고비, 택배비, 착불 택배비, 보관료, 특수작업, 오배송/누락 차감
   - 파일 업로드: 작업비, 입출고비, 택배비 (Google Drive)
   - 총액 자동 계산 (stCalculateTotal)

2. **필터** (관리자)
   - 정산년월, 화주사
   - 전달 조회, 일괄 전달, 일괄 삭제

3. **정산 목록 테이블**
   - 관리자: 체크박스, 화주사, 총액, 세금, 결제금액, 상세, 상태, 세금계산서, 관리
   - 화주사: 정산년월, 총액, 세금, 결제금액, 상세, 상태, 세금계산서, 정산확인

4. **모달**
   - 정산 상세 (stSettlementDetailModal)
   - 상태 변경 (stStatusChangeModal)
   - 명세서 미리보기 (stStatementPreviewModal)

### 전역 변수

```javascript
stCurrentRole, stCurrentCompany, stCurrentUsername
stInitialized, stSelectedFiles, stCurrentSettlementId
stCurrentStatusModalSettlementId
```

### 인증 헤더

```javascript
stGetUserHeaders() → {
  'X-User-Role': encodeURIComponent(role),
  'X-User-Name': encodeURIComponent(username),
  'X-Company-Name': encodeURIComponent(company)
}
```

## 7. 상태 흐름

```
대기 → 전달 → 정산확인 → 입금완료
  ↑      ↑         ↑
  │      │         └─ 화주사: 전달 상태에서만 정산확인 가능
  │      └─ 관리자: 일괄 전달 (bulk-update-status)
  └─ 관리자: 되돌리기 가능
```

- **일괄 전달**: `정산확인`, `입금완료` 상태는 변경하지 않음 (화주사 확인 보호)
- **화주사**: `전달` → `정산확인`만 가능

## 10. 코딩 시 참고사항

### 권한 처리

- `get_user_context()`: 헤더에서 role, username, company_name 추출
- 화주사: `company_name` 필터 자동 적용
- 관리자: `company_name` 파라미터로 필터링

### 화주사명 매칭

- `api.database.models.normalize_company_name(name)`
- `api.database.models.get_company_search_keywords(company_name)`
- companies.search_keywords, 별칭(alias) 지원

### PostgreSQL / SQLite 호환

- `USE_POSTGRESQL` 플래그
- `%s` (PostgreSQL) vs `?` (SQLite) placeholder
- `RealDictCursor` vs `sqlite3.Row`

### 에러 처리

- API: `jsonify({ success: False, message: ... })` + HTTP 상태 코드
- 프론트: `fetch` → `response.json()` → `data.success` 체크
